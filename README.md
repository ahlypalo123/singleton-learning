# Практика 4. Паттерн «Одиночка» и альтернатива через DI

Учебный Java-проект для кейса **«Обращения студентов»**. Пакет проекта: `ru.iubip`.

Цель проекта — показать два способа организации единой конфигурации и аудита уведомлений:

1. **Классический Singleton** с ленивой потокобезопасной инициализацией и глобальным доступом.
2. **Альтернатива без глобального доступа**: один экземпляр конфигурации и аудитора создаётся в composition root и передаётся бизнес-компонентам через конструктор.

## Стек

- Java 17
- Maven
- JUnit 5
- PlantUML для UML-диаграмм

## Реализованные варианты

### Вариант 1: Singleton

`ApplicationConfiguration` — потокобезопасный Singleton с ленивой инициализацией. Он читает настройки из `ConfigurationSource`; по умолчанию используется `EnvironmentConfigurationSource`, который берёт значения из переменных окружения:

- `NOTIFICATION_PROVIDER` — канал уведомлений, по умолчанию `email`.
- `AUDIT_ENABLED` — включён ли аудит, по умолчанию `true`.

`SingletonNotificationService` намеренно обращается к конфигурации и аудитору через глобальные статические точки доступа. Это делает зависимость скрытой: чтобы протестировать сервис с другой конфигурацией, приходится использовать специальные тестовые методы `setSourceForTests` и `resetForTests`.

### Вариант 2: DI и единый жизненный цикл

`ApplicationContainer` — ручной composition root. Он один раз загружает `ConfigurationData`, создаёт один `InMemoryNotificationAuditor` и передаёт эти зависимости в `InjectedNotificationService`.

Бизнес-компонент не знает, откуда пришла конфигурация, не читает файл/переменные окружения и не обращается к статическим объектам. В тестах можно просто передать нужный объект или заглушку.

## Тестирование

Запуск всех тестов:

```bash
mvn test
```

Покрытые сценарии:

1. Singleton-вариант требует явного сброса глобального состояния для независимости тестов.
2. Singleton-вариант позволяет подменить источник конфигурации только через статический тестовый hook.
3. Singleton-конфигурация безопасно инициализируется при параллельном доступе.
4. DI-вариант создаёт независимые сервисы без статического сброса.
5. DI-вариант подменяет источник конфигурации передачей зависимости.
6. DI-контейнер держит один экземпляр конфигурации, аудитора и сервиса в рамках жизненного цикла приложения.

## Инженерное сравнение

### Когда Singleton может быть оправдан

Singleton допустим для действительно общесистемных, технических и практически неизменяемых объектов: например, системных часов, реестра метрик, низкоуровневого адаптера к логгеру или конфигурации небольшого приложения. Важно, чтобы объект был потокобезопасным, не хранил изменяемое бизнес-состояние и не превращался в скрытый сервис-локатор.

### Риски Singleton в этом проекте

В кейсе уведомлений Singleton создаёт скрытые зависимости: `SingletonNotificationService` сам достаёт конфигурацию и аудитора. Из-за этого тест не видит зависимости через конструктор, а порядок тестов может начать влиять на результат. Если конфигурация уже была загружена, новая подмена источника не сработает без `resetForTests`. Такой сброс — дополнительная дисциплина, которую легко забыть, особенно при параллельном запуске тестов.

### Почему DI предпочтительнее

DI-вариант сохраняет требование «один экземпляр на приложение», но переносит управление жизненным циклом в `ApplicationContainer`. Сервис получает готовые зависимости и остаётся обычным объектом. Это улучшает читаемость архитектуры, упрощает замену in-memory аудитора на файловый или базовый и позволяет создавать независимые экземпляры в тестах.

### Как различается тестирование

В Singleton-варианте тесты вынуждены работать со статическим состоянием: настраивать источник через test-only API, сбрасывать конфигурацию и очищать singleton-аудитор после каждого теста. В DI-варианте тест создаёт нужные зависимости локально и передаёт их в конструктор, поэтому не нужен глобальный сброс, а тесты проще запускать в любом порядке.
